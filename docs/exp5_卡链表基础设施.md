# 实验五：卡链表基础设施

## 1. 本轮目标
本轮对应 Iteration 5.2，目标不是重写现有链表实现，而是把当前已经存在的链表基础设施正式确认为实验五的内存基础，并补齐实验五自己的基础设施说明与验证测试。

当前项目已经具备：
- `CardNode` 结点定义
- 数据层私有头指针管理
- 链表尾插
- 按卡号查找
- 删除结点
- 链表释放
- 清空后重新插入

因此实验五后续开发应直接复用这些能力，而不是重新搭一套链表底座。

## 2. 当前链表基础设施
### 2.1 结点结构
链表结点定义在 [card_storage.h](../include/card_storage.h) 中：
- 数据域：`Card cardData`
- 指针域：`struct CardNode *pNext`

### 2.2 头指针管理
当前链表头指针由数据层私有静态变量维护：
- `g_pCardListHead`

该设计意味着：
- 链表的生命周期统一由数据层管理
- 表示层和业务层不直接持有头指针
- 后续实验五继续沿用“业务层调用数据层能力”的模式

### 2.3 已具备的链表能力
当前数据层已具备：
1. 空链表初始化状态
   - 程序启动后头指针为空
   - `dataCleanup()` 可将链表恢复为空状态
2. 链表尾插
   - `dataAddCard()` 在空链表时插入首结点
   - 在非空链表时遍历至尾结点完成尾插
3. 按卡号查找
   - `dataFindCardByName()` 可按卡号遍历查找
4. 删除结点
   - `dataDeleteCardByName()` 支持删除头/中/尾结点
5. 链表释放与重建
   - `dataCleanup()` 负责释放全部结点
   - 释放后可再次正常插入新结点

## 3. 5.2 的结论
Iteration 5.2 的结论是：
- 实验五后续直接复用当前链表基础设施
- 不再重复做“结构体数组迁移到链表”的工作
- 5.3 以后新增的是查询扩展能力，而不是链表底座重写

也就是说，后续实验五的重点应转向：
- 如何在现有链表上完成模糊查询
- 如何把模糊查询结果以列表方式展示出来
- 如何建立实验五自己的测试与验收材料
