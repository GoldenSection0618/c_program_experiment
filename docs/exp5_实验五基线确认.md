# 实验五：基线确认

## 1. 本轮目标
本轮对应 Iteration 5.1，只做实验五开发基线确认，不修改现有功能代码，不重构实验一至实验四已经完成的主路径。

基线确认的核心结论是：当前项目已经具备“链表 + 文本文件持久化 + 精确查询”的实验五前置资产，后续实验五应在此基础上增量补齐“模糊查询 + 实验五专属测试与说明”，而不是重复做一次“结构体数组切换到链表”的迁移。

## 2. 已确认可复用资产

### 2.1 菜单与表示层
当前主菜单仍保留实验五需要继续使用的两个入口：
- 菜单 `1`：添加卡
- 菜单 `2`：查询卡

当前表示层已经具备：
- 添加卡输入采集
- 查询卡输入采集
- 成功/失败信息输出
- 卡信息表格展示

因此实验五后续不需要重做菜单骨架，只需要在现有交互基础上补足模糊查询入口与展示逻辑。

### 2.2 数据模型
当前 [model.h](../include/model.h) 中的 `Card` 结构已经稳定，字段覆盖：
- 卡号
- 密码
- 状态
- 开卡时间
- 截止时间
- 累计金额
- 最后使用时间
- 使用次数
- 余额
- 删除标识

该结构满足实验五继续使用的需要，无需重新定义另一套卡信息结构。

### 2.3 链表基础设施
当前 [card_storage.h](../include/card_storage.h) 已定义 `CardNode`，数据层已经具备：
- 头指针管理
- 链表尾插
- 按卡号查找
- 删除结点
- 链表释放
- 私有计数维护

因此实验五后续开发不需要重新搭建链表基础设施，只需在现有链表能力之上增加“模糊遍历匹配”。

### 2.4 文件持久化能力
当前 [repository.c](../src/data/repository.c) 已具备以下能力：
- `saveCard`：保存单条卡记录到文本文件
- `readCard`：从文本文件读取全部卡记录并恢复到链表
- `praseCard`：解析单条文本记录
- `stringToTime`：恢复时间字段
- `getCardCount`：统计记录数量
- `isCardExists`：判断卡号是否存在
- `updateCard`：重写文件中的指定卡记录

当前文本持久化路径仍为：`data/cards.txt`。

### 2.5 当前业务主线
当前业务层已具备实验五可直接复用的两条主线：

#### 添加卡主线
当前 `bizAddCard` 已按以下顺序组织：
1. 输入标准化与合法性校验
2. 读取文件并重建链表
3. 重复卡号校验
4. 构造完整 `Card`
5. 插入链表
6. 写回文件
7. 返回结果给表示层

#### 精确查询主线
当前 `bizQueryCard` 已按以下顺序组织：
1. 输入标准化与合法性校验
2. 查询前读取文件并重建链表
3. 按卡号精确匹配
4. 返回结果给表示层展示

## 3. 已确认无需再做的内容

### 3.1 不需要再做数组迁移
虽然 `model.h` 中仍保留 `MAX_CARD_COUNT`，但当前主路径已经不是结构体数组实现。

基线确认结果表明：
- 当前代码库里没有结构体数组版卡管理的主实现路径
- 当前卡管理运行时组织方式已经是链表
- 实验五后续不需要再做一次“数组切换到链表”的迁移

### 3.2 不需要推翻现有三层结构
当前项目三层职责已经基本清晰：
- 表示层：菜单输入输出与结果展示
- 业务逻辑层：添加卡、查询卡规则与流程协调
- 数据存储层：文本文件读写与链表装载

实验五后续应继续沿用该结构，不应把逻辑重新塞回 `main` 或菜单函数。

## 4. 已确认的实验五缺口
当前项目距离实验五要求的主要缺口只有以下几项：

### 4.1 缺少模糊查询能力
当前只有：
- `bizQueryCard(const char *cardNameInput, Card *queriedCard)`

这是一条精确查询接口，尚不支持：
- 输入关键字
- 遍历链表匹配“卡号包含关键字”的多条记录
- 返回多条查询结果
- 模糊查询无结果提示

### 4.2 缺少模糊查询展示能力
当前表示层展示组件只支持：
- 单张卡摘要展示
- 单张卡查询结果展示

尚未具备：
- 多结果列表展示
- 模糊查询结果计数或空结果提示的专用输出

### 4.3 缺少实验五专属测试套件
当前测试目录已覆盖：
- `exp3`：文件持久化、添加卡、精确查询
- `exp4`：链表删除操作

尚未建立：
- `exp5` 测试目录
- 模糊查询验收用例
- 实验五整体验收批量套件
