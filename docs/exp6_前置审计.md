# 实验六：前置审计

## 1. 审计目标
本轮对应实验六的迭代 0，只确认进入“上机 / 下机结算”开发前的基础条件是否满足，不在本轮修改业务功能。

## 2. 当前可直接复用的基础
### 2.1 菜单入口已存在
当前主菜单中已经保留：
- `3. 上机`
- `4. 下机`

表示层分发也已经接通到业务层入口，因此实验六不需要重新搭建菜单骨架，只需要把占位入口替换成真实业务流程。

### 2.2 卡文件读写稳定
当前项目已经完成实验三的文本文件持久化方案，卡信息文件仍为：
- `data/cards.txt`

数据层已经具备这些能力：
- `dataLoadCards()`：从文件恢复全部卡信息到内存链表
- `dataSaveCard()`：追加保存单张卡信息
- `dataUpdateCard()`：修改链表节点后重写文件
- `dataGetCardCount()`：统计卡文件中的有效记录数

因此实验六不需要重做卡文件读写底座。

### 2.3 卡链表装载稳定
当前内存载体仍然是链表，数据层已经具备：
- `dataAddCard()`：插入新卡结点
- `dataDeleteCardByName()`：删除指定卡结点
- `dataQueryCardByName()`：按卡号精确查询
- `dataQueryCardsByKeyword()`：按关键字查询
- `dataCleanup()`：释放当前卡链表

这说明实验六可以继续沿用“文件 -> 链表 -> 业务处理”的主路径，无需回退到数组方案。

### 2.4 精确查卡稳定
当前精确查询主线已经在实验五完成：
1. 读取输入卡号
2. `dataLoadCards()` 从文件恢复链表
3. `dataQueryCardByName()` 在链表中精确匹配
4. 输出结果或失败提示

实验六的“上机 / 下机”都需要先按卡号、密码定位目标卡，因此这一能力可直接复用。

### 2.5 卡状态字段可读可写
当前 `Card` 结构已经包含并使用这些状态值：
- `0`：未上机
- `1`：正在上机
- `2`：已注销
- `3`：失效

这满足实验六对“上机前判断 / 下机后恢复状态”的基础要求。

### 2.6 时间类型与时间格式化工具可用
当前项目已具备：
- `time_t` 时间字段
- `localtime + strftime` 格式化输出
- `stringToTime()` 文本时间反序列化

因此实验六可以直接在现有体系上增加：
- 上机时间记录
- 下机时间记录
- 计费时长计算
- 结算结果展示

## 3. 当前未完成、必须进入实验六实现的内容
### 3.1 上机 / 下机业务仍是占位实现
当前业务层中：
- `bizStartBilling()`
- `bizStopBilling()`

仍然只输出占位提示，没有真正完成：
- 卡号 + 密码校验
- 卡状态判断
- 上机记录创建
- 下机结算更新

这说明实验六的核心业务尚未开始实现。

### 3.2 消费记录模型虽已定义，但尚未形成主线能力
当前 `model.h` 里已经有：
- `Billing`
- `LogonInfo`
- `SettleInfo`

但数据层尚未建立完整的消费记录持久化与查询更新主线，实验六必须新增：
- 消费记录文件
- 读取全部消费记录
- 追加未结算消费记录
- 查找未结算记录
- 更新已结算记录
- 释放消费记录内存结构

### 3.3 计费规则尚未落地
当前项目还没有实验六所需的真正计费函数，必须在后续迭代中补上：
- 输入：上机时间、下机时间、费率
- 输出：时长、消费金额

### 3.4 菜单 3 / 4 的表示层交互未实现
当前菜单 3 / 4 已接入，但没有：
- 输入卡号、密码
- 输出上机成功结果
- 输出下机成功结果
- 输出失败提示

这部分需要在实验六迭代中从占位入口升级为真实交互流程。

## 4. 结论
实验六“前置审计”结论如下：
- 菜单 3 / 4 已存在，可直接扩展
- 卡文件读写稳定，可继续复用
- 卡链表装载稳定，可继续复用
- 精确查卡稳定，可继续复用
- 卡状态字段已具备，可直接用于上机 / 下机状态转换
- 时间类型与时间工具已具备，可直接用于记录和结算
- 当前真正缺失的是：消费记录持久化、上机业务、下机结算、计费规则

因此，实验六后续开发不需要返工实验三到实验五的底座，应直接进入：
1. 消费记录数据模型与文件存储
2. 上机业务主流程
3. 下机结算主流程
4. 错误分支与脏数据保护
